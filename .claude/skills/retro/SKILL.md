# Retro Skill

세션 회고 + 스킬/에이전트 분석 + 개선 반영을 한 번에 수행하는 워크플로우 스킬.

## When to Use
- 세션 마무리 시 작업 정리가 필요할 때
- 스킬/에이전트가 기대대로 동작하지 않았을 때
- 주기적으로 워크플로우를 점검하고 싶을 때

## Workflow

```
[1. 세션 분석]
     |
     v
[2. 스킬/에이전트 분석]
     |
     v
[3. 문제점 도출 + 개선안 제안]
     |
  사용자 확인
     |
     v
[4. 개선 반영]
     |
     v
[5. 메모리 업데이트]
     |
     v
[6. 회고 보고]
```

## Step 1: 세션 분석

현재 세션에서 수행한 작업을 정리한다.

1. Linear MCP로 오늘 상태가 변경된 이슈 조회
2. git log로 오늘의 커밋/PR 확인
3. 세션 중 발생한 에러, 재시도, 의사결정 정리

출력 형식:
```markdown
### 세션 요약
- 완료: {이슈 목록}
- PR: {PR 목록}
- 의사결정: {주요 결정 사항}
- 문제 발생: {에러/재시도 이력}
```

## Step 2: 스킬/에이전트/인프라 분석

`.claude/skills/`, `.claude/agents/`, 그리고 주요 인프라 파일을 읽고 분석한다.

### 분석 대상
- `.claude/skills/` — 모든 스킬 파일
- `.claude/agents/` — 모든 에이전트 파일
- `.github/workflows/` — GitHub Actions 워크플로우
- `scripts/` — 빌드/생성 스크립트 (주요 파일만)
- `CLAUDE.md`, `.claude/plan.md` — 프로젝트 설정

### 에이전트 분석 기준
| 항목 | 확인 내용 |
|------|----------|
| Context 정확성 | 프로젝트 현재 상태와 일치하는가 (기술 스택, 디렉토리 구조 등) |
| Input/Output 명확성 | 입력/출력 포맷이 구체적이고 다음 에이전트가 바로 사용 가능한가 |
| Constraints 실효성 | 실제 세션에서 지켜졌는가, 누락된 제약이 있는가 |
| Workflow 효율성 | 불필요한 단계가 있는가, 병목이 있는가 |

### 스킬 분석 기준
| 항목 | 확인 내용 |
|------|----------|
| 경로/설정 최신성 | 하드코딩된 경로, 설정이 현재 프로젝트와 일치하는가 |
| 에러 핸들링 | 실제 발생한 에러가 Error Handling에 커버되는가 |
| 사용자 경험 | 불필요한 확인 단계, 누락된 확인 단계가 있는가 |
| 도구 호출 정확성 | MCP 도구명, 파라미터가 실제 동작과 일치하는가 |

### 인프라 분석 기준
| 항목 | 확인 내용 |
|------|----------|
| 워크플로우 정상 동작 | GitHub Actions cron/workflow_dispatch가 성공하는가 |
| 스크립트 안정성 | 파싱 실패, API 에러 핸들링이 충분한가 |
| 환경변수 일관성 | .env.example, Vercel, GitHub Secrets 간 동기화 |
| 외부 API 의존성 | API 변경/과금 이슈, 폴백 전략이 있는가 |

## Step 3: 문제점 도출 + 개선안 제안

분석 결과를 정리하여 사용자에게 제안한다.

```markdown
### 발견된 문제점

| # | 구분 | 파일 | 문제 | 개선안 | 심각도 |
|---|------|------|------|--------|--------|
| 1 | Agent | developer.md | ... | ... | High |
| 2 | Skill | standup/SKILL.md | ... | ... | Medium |
```

**사용자 확인**: AskUserQuestion으로 어떤 개선을 반영할지 선택받는다.

```
AskUserQuestion(
  questions=[{
    "question": "어떤 개선을 반영할까요?",
    "header": "개선 반영",
    "multiSelect": true,
    "options": [
      {"label": "#{번호}: {요약}", "description": "{상세}"},
      ...
    ]
  }]
)
```

## Step 4: 개선 반영

사용자가 선택한 개선사항을 Edit 도구로 직접 반영한다.

- 에이전트 파일 수정: `.claude/agents/*.md`
- 스킬 파일 수정: `.claude/skills/*/SKILL.md`
- CLAUDE.md 업데이트 (필요 시)
- plan.md 업데이트 (필요 시)

수정 후 변경된 파일 목록을 출력한다.

## Step 5: 메모리 업데이트

세션에서 발견된 안정적인 패턴과 결정사항을 메모리에 반영한다.

대상 파일: `/Users/sunny/.claude/projects/-Users-sunny-deep-thought-workspace-sunny-blog/memory/MEMORY.md`

업데이트 원칙:
- 이미 있는 정보는 최신화 (덮어쓰기)
- 세션 한정 임시 정보는 저장하지 않음
- 검증된 패턴만 기록 (추측 금지)

## Step 6: 회고 보고

최종 보고서를 출력한다.

```markdown
## Retro 완료

### 세션 성과
| 지표 | 값 |
|------|-----|
| 완료 이슈 | {N}개 |
| 생성 PR | {N}개 |
| 파이프라인 실행 | {N}회 |

### 개선 반영
| 파일 | 변경 내용 |
|------|----------|
| {파일} | {요약} |

### 다음 세션 권장 작업
1. {작업 1}
2. {작업 2}

### 워크플로우 건강도
- 에이전트: {OK/개선필요} — {요약}
- 스킬: {OK/개선필요} — {요약}
- Linear 이슈: {잔여 이슈 수}개 Todo
```

## Error Handling

| 상황 | 대응 |
|------|------|
| 분석할 에이전트/스킬 파일이 없음 | 파일 목록 출력 후 종료 |
| 사용자가 개선 없음 선택 | Step 4 건너뛰고 메모리 업데이트만 수행 |
| 메모리 파일 접근 불가 | 경로 확인 안내 |

## Quick Reference

```
/retro
  -> 세션 분석 (Linear 이슈 + git log)
  -> 스킬/에이전트 파일 전수 분석
  -> 문제점 도출 + 개선안 제안
  -> 사용자 선택 -> 개선 반영
  -> 메모리 업데이트
  -> 회고 보고
```
